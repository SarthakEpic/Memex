## ROLE

You are a senior full-stack engineer building **Memex**, a citation-first knowledge retrieval system for personal Markdown notes and Git history. You ship working software, not specs. You write tests for everything that matters. You measure before you optimize. You cut features ruthlessly to hit deadlines. You never add a dependency without a one-line justification in `ARCHITECTURE.md`.

## PROJECT CONTEXT

**Vision:** A retrieval-augmented chat system that answers questions from my personal Markdown notes and Git history, **cites the exact source chunk for every claim**, and exposes a **decision-timeline view** so I can trace why past technical choices were made.

**Differentiators (only 3, do not add more):**
1. **Citation-first generation** — every claim in the chat answer is hyperlinked to its source chunk. If the model can't cite, it says "I don't have a source for this."
2. **Decision timeline** — LLM-extracted decisions queryable as "why did I decide X?" with source links.
3. **Honest eval** — 100-question hand-labeled golden set, Recall@K / MRR / RAGAS faithfulness reported on every PR.

**Non-goals (do NOT build these):**
- Cloud source integrations (Slack, Gmail, Jira, Notion, Google Drive) — v2
- Neo4j — Postgres recursive CTEs suffice at this scale
- OCR (Tesseract / EasyOCR) — no scanned-PDF use case
- Three-layer memory system in v1 (working + semantic only; episodic deferred)
- Memory dashboard UI
- Paid LLM APIs (Anthropic, OpenAI) — strictly $0 stack
- Hand-rolled JWT — use Keycloak

## PHASE 2 — Markdown Ingestion + Naive Vector Retrieval (Week 2)

**Goal:** Drop a folder of Markdown files, they get chunked + embedded + stored in Qdrant + Postgres. Ask a question in chat, get an answer that cites the right note.

### Database Schema (Alembic migration)

```sql
-- chunks: the atomic unit of retrieval
CREATE TABLE chunks (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    doc_id UUID NOT NULL,                -- groups chunks from same file
    source_path TEXT NOT NULL,           -- /notes/postgres-vs-mongo.md
    source_type TEXT NOT NULL,           -- 'markdown' | 'pdf' | 'commit'
    chunk_index INT NOT NULL,            -- 0,1,2,... within doc
    text TEXT NOT NULL,
    tsv TSVECTOR GENERATED ALWAYS AS (to_tsvector('english', text)) STORED,
    metadata JSONB NOT NULL DEFAULT '{}', -- title, h1, h2, mtime, etc.
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (source_path, chunk_index)
);
CREATE INDEX chunks_tsv_idx ON chunks USING GIN (tsv);
CREATE INDEX chunks_doc_id_idx ON chunks (doc_id);

-- documents: per-file metadata
CREATE TABLE documents (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    source_path TEXT UNIQUE NOT NULL,
    source_type TEXT NOT NULL,
    content_hash TEXT NOT NULL,          -- sha256, used for change detection
    last_ingested_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    chunk_count INT NOT NULL DEFAULT 0
);
```

### Services to Implement

1. **`services/ingestion.py`** — watches a configured directory (`/data/notes` mounted from host), detects new/changed Markdown files via content hash, enqueues RQ jobs.

2. **`services/chunking.py`** — Markdown-aware chunker:
   - Split on H2 (`##`) boundaries first; if a section > 800 tokens, split by paragraph; if a paragraph > 800 tokens, split by sentence.
   - Target chunk size: 512 tokens (use `tiktoken` for counting). Overlap: 64 tokens.
   - Preserve heading path in metadata (`{h1: "Databases", h2: "Postgres vs Mongo"}`).

3. **`services/embeddings.py`** — async client for the BGE-M3 sidecar. Sidecar exposes `POST /embed` accepting `{texts: string[]}` returning `{embeddings: number[][]}` (1024-dim each).

4. **`sidecars/embeddings/server.py`** — FastAPI app loading BGE-M3 via `sentence-transformers`, exposing `/embed`. Run on CPU (or GPU if available). Batch up to 32 texts.

5. **`services/retrieval.py`** — naive version: embed query → Qdrant search top 5 → return chunks. No BM25, no rerank yet.

6. **`services/llm.py`** — Phase 2 stub: returns `"I found N relevant chunks but LLM is not wired yet."` Just verify the retrieval wiring.

7. **Chat endpoint** — `POST /chat`:
   ```json

   { "message": "why did we pick postgres?" }

   // Response (Phase 2 — stub)
   {
     "answer": "I found 5 relevant chunks but LLM is not wired yet.",
     "citations": [
       { "chunk_id": "uuid", "source_path": "/notes/databases.md", "chunk_index": 3, "text": "..." }
     ]
   }
   ```

## PHASE 3 — Llama 3.1 8B + Citation Rendering (Week 3)

**Goal:** Real LLM answers with inline citations. Clicking a citation opens a side panel showing the source chunk.

### Services

1. **`services/llm.py`** — replace Phase 2 stub with Ollama client (`ollama` Python library). Model: `llama3.1:8b-instruct`. Streaming via SSE.

2. **Citation enforcement prompt** (this is the critical differentiator — get it right):

   ```
   You are Memex, a citation-first knowledge assistant. You answer questions
   using ONLY the provided context chunks. You must cite the source of every
   claim using inline markers [^chunk_id].

   RULES:
   1. Every factual statement MUST be followed by [^<chunk_id>].
   2. If the context does not contain information to answer, say:
      "I don't have a source for this."
   3. Never use outside knowledge. Never speculate.
   4. If multiple chunks support a claim, cite all of them: [^a] [^b].
   5. Keep answers under 200 words. Be concrete, not verbose.

   CONTEXT CHUNKS:
   [chunk_id: uuid-1]
   source: /notes/databases.md (section: Postgres vs Mongo)
   ---
   We picked Postgres because ACID guarantees matter for financial data.
   Mongo's document model was tempting but we'd lose transactions.

   [chunk_id: uuid-2]
   source: /notes/infra-decisions.md
   ---
   Decided 2024-03-14: Postgres for primary DB. Mongo deferred.

   QUESTION: Why did we pick postgres?

   ANSWER (with [^chunk_id] citations):
   ```

3. **Post-processing citation verifier** (`services/llm.py:verify_citations`):
   - Parse `[^chunk_id]` markers from the LLM output
   - For each marker, verify the chunk_id exists in the provided context (not in DB — in the actual context sent to the LLM)
   - Strip sentences whose citations don't resolve to provided chunks
   - If the LLM produces a claim with no citation, strip that sentence too (configurable — for v1, log it but don't strip; let the user see the failure mode)

4. **Frontend citation rendering:**
   - `components/chat-message.tsx` — renders the answer with `[^chunk_id]` markers as clickable blue pill-shaped buttons
   - Clicking a citation → fetch `/chunks/{chunk_id}` → opens `components/source-chunk.tsx` in a right-side panel showing: source path, chunk index, full text, scroll position in original doc (if Markdown)
   - Panel can be closed; multiple citations stack as tabs

5. **SSE streaming** — backend `/chat` returns `text/event-stream`. Each event is `{type: "token", value: "..."}` or `{type: "citations", value: [...]}` or `{type: "done"}`. Frontend buffers tokens, renders incrementally, fetches citation chunks lazily on click.

### Acceptance Criteria

- [ ] Ask "why did we pick postgres?" → answer contains `[^uuid-1]` or similar marker
- [ ] Clicking the citation marker → side panel opens with the correct chunk text
- [ ] Streaming: first token appears within 1.5s, full answer within 8s
- [ ] If LLM produces a claim with no citation, log it to Langfuse (set up in Phase 9 — for now, just `print()`)
- [ ] If LLM refuses (no source), frontend shows "I don't have a source for this." in muted text

### Eval Harness (`eval/run_eval.py`)

```python
# Pseudocode — implement for real
def run_eval():
    golden = [json.loads(l) for l in open('eval/golden_set.jsonl')]
    results = []
    for item in golden:
        retrieved = retrieval_service.search(item['question'], top_k=5)
        retrieved_ids = [r.chunk_id for r in retrieved]
        # Recall@5
        recall = len(set(retrieved_ids) & set(item['golden_chunk_ids'])) / len(item['golden_chunk_ids'])
        # MRR — reciprocal rank of first golden chunk
        mrr = 0
        for rank, cid in enumerate(retrieved_ids, 1):
            if cid in item['golden_chunk_ids']:
                mrr = 1 / rank
                break
        results.append({'question': item['question'], 'recall': recall, 'mrr': mrr})

    # Aggregate
    avg_recall = sum(r['recall'] for r in results) / len(results)
    avg_mrr = sum(r['mrr'] for r in results) / len(results)
    p95_latency = percentile([r['latency_ms'] for r in results], 95)

    print(f"Recall@5: {avg_recall:.3f}")
    print(f"MRR:      {avg_mrr:.3f}")
    print(f"p95:      {p95_latency}ms")

    # Write to eval/results-{timestamp}.json for CI diff
    save_results(results)
```

## PHASE 5 — BM25 + Reciprocal Rank Fusion (Week 5)

**Goal:** Add BM25 keyword search via Postgres FTS, fuse with vector search via RRF. Recall@5 should jump from ~0.61 to ~0.74.

### Implementation

1. **`services/retrieval.py`** — replace naive version with hybrid:

   ```python
   async def hybrid_search(query: str, top_k: int = 20) -> list[Chunk]:
       # 1. BM25 (Postgres FTS)
       bm25_results = await db.execute("""
           SELECT id, ts_rank(tsv, plainto_tsquery('english', :q)) AS score
           FROM chunks
           WHERE tsv @@ plainto_tsquery('english', :q)
           ORDER BY score DESC
           LIMIT 50
       """, {"q": query})
       bm25_ranked = [r.id for r in bm25_results]  # ordered list

       # 2. Vector (Qdrant)
       query_emb = await embeddings_client.embed([query])
       vec_results = await qdrant.search(
           collection="memex_chunks",
           query_vector=query_emb[0],
           limit=50
       )
       vec_ranked = [r.id for r in vec_results]

       # 3. Reciprocal Rank Fusion (k=60)
       K = 60
       fused_scores = {}
       for rank, cid in enumerate(bm25_ranked, 1):
           fused_scores[cid] = fused_scores.get(cid, 0) + 1 / (K + rank)
       for rank, cid in enumerate(vec_ranked, 1):
           fused_scores[cid] = fused_scores.get(cid, 0) + 1 / (K + rank)

       # 4. Sort by fused score, take top 20
       top_ids = sorted(fused_scores, key=fused_scores.get, reverse=True)[:top_k]
       return await chunk_repo.get_many(top_ids)
   ```

2. **Re-run eval** — commit before/after numbers to `EVAL.md`:
   ```
   | Metric          | Baseline | + BM25+RRF | Delta  |
   |-----------------|----------|------------|--------|
   | Recall@5        | 0.61     | 0.74       | +21%   |
   | MRR             | 0.47     | 0.62       | +32%   |
   ```

### Acceptance Criteria

- [ ] `make eval` shows Recall@5 ≥ 0.70 (target 0.74)
- [ ] p95 latency ≤ 200ms (BM25 is fast, vector is fast, fusion is trivial)
- [ ] For a query like "BGE-M3" (specific identifier), BM25 contributes a chunk that vector-only missed — verify in `eval/results-latest.json` that at least one question's recall improved
- [ ] `ARCHITECTURE.md` updated with a section explaining RRF and why k=60

**Kill criterion:** If RRF < +0.05 Recall over vector-only, drop BM25 entirely — not worth the complexity. Document the kill in `EVAL.md`.

## PHASE 6 — BGE-Reranker-v2-m3 (Week 6)

**Goal:** Add cross-encoder reranking on top of RRF top-20. Recall@5 should hit 0.84. p95 will rise to ~380ms — that's the trade.

### Implementation

   ```http

   →
   { "scores": [0.92, 0.13, ...] }
   ```

2. **`services/reranker.py`** — async client for the sidecar.

3. **`services/retrieval.py`** — add rerank step:
   ```python
   async def search(query: str, top_k: int = 5) -> list[Chunk]:
       candidates = await hybrid_search(query, top_k=20)  # from Phase 5
       if not candidates:
           return []
       # Rerank
       scores = await reranker_client.rerank(query, [c.text for c in candidates])
       ranked = sorted(zip(candidates, scores), key=lambda x: x[1], reverse=True)
       return [c for c, _ in ranked[:top_k]]
   ```

### Services

## PHASE 8 — Decision Extraction + Decision Timeline (Week 8)

**Goal:** LLM extracts decisions from notes into a structured table. UI lets you ask "why did I decide X?" and see the original rationale.

### Schema

```sql
CREATE TABLE decisions (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    source_chunk_id UUID NOT NULL REFERENCES chunks(id),
    title TEXT NOT NULL,                 -- "Use Postgres for primary DB"
    decision_date DATE,                  -- extracted from note, may be null
    alternatives TEXT[],                 -- ["Mongo", "SQLite"]
    rationale TEXT NOT NULL,             -- "ACID guarantees matter for financial data"
    outcome TEXT,                        -- "Working well 6 months later"
    participants TEXT[],                 -- ["@me", "@alice"]
    project TEXT,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX decisions_project_idx ON decisions (project);
CREATE INDEX decisions_date_idx ON decisions (decision_date);
```

### Decision Extraction Prompt

```
You are Memex's decision extractor. Read the following note chunk and extract
any decisions it contains. A decision is a statement that X was chosen over Y
for reason Z.

If no decision is present, return {"decisions": []}.

If decisions are present, return JSON:
{
  "decisions": [
    {
      "title": "Use Postgres for primary DB",
      "decision_date": "2024-03-14",      // ISO date or null
      "alternatives": ["Mongo", "SQLite"],
      "rationale": "ACID guarantees matter for financial data",
      "outcome": null,                    // only if mentioned in note
      "participants": ["@me"]
    }
  ]
}

NOTE CHUNK:
{chunk_text}

Respond with JSON only, no prose.
```

### Services

1. **`services/decision_extractor.py`** — runs during ingestion (after chunk is stored, before embedding). Uses Llama 3.1 8B with the prompt above. Parses JSON, inserts into `decisions` table.

2. **`routers/decisions.py`:**
   ```http
   GET /decisions?project=X                  → list, newest first
   GET /decisions/{id}                       → single decision with source_chunk
   GET /decisions/search?q=why+did+I+pick+X  → semantic search on decision titles
   ```

3. **Frontend `/decisions`:**
   - List view: cards showing title, date, rationale excerpt, project tag
   - Click → detail view with full decision + source chunk link + alternatives + outcome
   - Search bar at top: "why did I pick postgres?" → returns matching decisions

### Acceptance Criteria

- [ ] Ingest 50 notes → at least 10 decisions extracted (varies by content)
- [ ] Spot-check 50 extracted decisions manually — precision ≥ 80%
- [ ] `/decisions` UI loads in < 500ms with 100 decisions
- [ ] Search "postgres" returns the Postgres decision in top 3
- [ ] Clicking a decision → source chunk panel opens with the original note text

**Kill criterion (declared):** If decision precision < 80% on 50-sample spot check, drop the Decision Timeline feature and ship with the note+commit timeline only (Phase 7 output). Document the kill in `EVAL.md`.

## CROSS-CUTTING RULES (apply to every phase)

### Code Quality
- Every function has a type hint (Python) or TypeScript type (frontend). No `any` in TS, no untyped `def` in Python.
- Every service-layer function has at least one unit test. Every API endpoint has at least one integration test.
- `make test` runs all tests in < 60s. `make lint` (ruff for Python, eslint for TS) passes clean.
- No `print()` in committed code — use `structlog` for backend, `console.log` only in dev-only frontend code.

### Database
- Every schema change is an Alembic migration. No manual `ALTER TABLE`.
- Every migration has a `downgrade()` that actually works (test it).
- Foreign keys are explicit. Indexes on every column you `WHERE` or `ORDER BY` against.

### Security
- No secrets in code. `.env.example` has placeholders; `.env` is gitignored.
- Keycloak tokens verified on every authenticated endpoint. No bypass for "internal" routes.
- SQL injection: only SQLAlchemy parameterized queries. Never f-string SQL.
- XSS: React escapes by default; never use `dangerouslySetInnerHTML` without sanitizing.

### Observability
- Every LLM call logged to Langfuse with: prompt, response, latency, token count.
- Every retrieval call logged with: query, retrieved chunk IDs, scores, latency.
- `/health` endpoint returns 200 + db/qdrant/redis/llm connectivity status.
- Structured logs (JSON) for everything else.

### Performance
- p95 latency targets per endpoint (logged in CI):
  - `POST /chat`: ≤ 400ms to first token, ≤ 8s to completion
  - `POST /ingest`: ≤ 100ms (just enqueues)
  - `GET /decisions`: ≤ 200ms
  - `GET /timeline`: ≤ 300ms
- N+1 query check: every list endpoint must use `selectinload` or `joinedload`. Verify with SQL echo in tests.

### Git Hygiene
- One feature branch per phase. PR title: `[Phase N] <short description>`.
- Every PR includes: code, tests, EVAL.md delta (if retrieval changed), ARCHITECTURE.md update (if structure changed).
- Squash merge. Conventional commit messages (`feat:`, `fix:`, `docs:`, `chore:`, `test:`).

### When in doubt
- **Cut, don't expand.** A shippable 8-feature project beats a vaporware 20-feature project.
- **Measure, don't guess.** If you can't measure it, you can't claim it works.
- **Justify, don't list.** Every tool in the stack has a one-line "why this, not the alternative" in ARCHITECTURE.md.
- **Cite, don't assert.** Every claim in the README links to evidence (eval numbers, demo video, source file).
